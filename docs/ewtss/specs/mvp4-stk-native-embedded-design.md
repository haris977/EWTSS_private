# MVP4 — STK-Native Embedded Authoring (Option D2) — Design Spec

**Date:** 2026-04-22
**Status:** Draft for user review
**Builds on:** [`v2-tech-stack-archive.md`](./v2-tech-stack-archive.md) — §14 (Options D1/D2) and §22 (CesiumJS vs STK native display) already documented the trade-offs between Cesium-based and STK-native visualisation. MVP4 picks the **D2** variant and builds a free-form authoring shell around it.

---

## 1. Goal

Prove that the EWTSS scenario-generator workflow — authoring + computation + visualisation — can be delivered as a **single Windows desktop application** with STK Engine as the unified backbone, replacing the MVP1–MVP3 shape of Python FastAPI + Angular + Cesium.

MVP4's success is narrow: a C# dev can build a scenario from scratch containing **Aircraft, Facility, AreaTarget, Sensor, CoverageDefinition, and FigureOfMerit**, compute it, visualise it in the embedded STK globe, save it as `.sc` or `.vdf`, and re-open it.

The reference scenario is a CZML export the user provided during brainstorming (STK's `Scenario34`) containing exactly those six entity types. It will be committed as `mvp4/docs/acceptance/Scenario34.czml` during MVP4 Phase 1 so integration tests have a fixed reference.

---

## 2. Key decisions (from brainstorming)

| # | Decision | Alternative rejected | Why |
|---|---|---|---|
| 1 | Free-form authoring of scenarios (not load-and-tweak) | Load preset `.sc`; template-wizard | "Embed STK engine for planning" only has teeth if the operator can create entities from scratch |
| 2 | C# WPF owns STK end-to-end (Option D2) | D1 (Python computes, C# visualises); hybrid C# + thin Python REST | STK's COM API was designed for C#; keeps the MVP surface small; property-panel bindings require round-trip-free access to COM objects |
| 3 | No non-STK analytics (no FOM heatmap export, no DRS overlays) | MVP3-parity with FOM heatmap over Cesium; DRS Kafka overlays | MVP4 validates the architectural shift; re-implementing MVP3's analytics would dilute the thesis |
| 4 | Entity set: Aircraft, Facility, AreaTarget, Sensor, CoverageDefinition, FigureOfMerit | Just 2 aircraft + sensors (too narrow); full Insight breadth with satellites/comms (too wide) | Matches the attached reference CZML; enough to exercise compute + rendering; bounded property-panel surface |

## 3. Architecture overview

MVP4 is a **single Windows desktop process**. No servers, no Kafka, no Python, no DB.

```
┌─────────────────────────────────────────────────────┐
│  Sg.Mvp4.App.exe   (C# WPF, .NET 8)                │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │  MainWindow (XAML)                          │   │
│  │                                             │   │
│  │  ┌────────┬─────────────────────┬────────┐  │   │
│  │  │ Object │  Embedded STK       │ Prop   │  │   │
│  │  │ Tree   │  Globe (ActiveX)    │ Panel  │  │   │
│  │  │        │  + 2D Map tab       │        │  │   │
│  │  └────────┴─────────────────────┴────────┘  │   │
│  │            Toolbar + native Timeline        │   │
│  └─────────────────────────────────────────────┘   │
│                    │                                │
│                    │  COM automation                │
│                    ▼                                │
│           ┌────────────────────┐                   │
│           │ IAgStkObjectRoot   │  ◄─ single in-    │
│           │ (AGI.STKObjects 12)│    stance owned   │
│           └────────┬───────────┘    by the app     │
│                    │                                │
│                    ▼                                │
│           ┌────────────────────┐                   │
│           │ STK 12 Engine      │  ◄─ out-of-proc   │
│           │ (COM server)       │    COM server     │
│           └────────────────────┘                   │
│                                                    │
│  Scenario files:  C:\EWTSS\mvp4\scenarios\*.sc    │
└────────────────────────────────────────────────────┘
```

**Invariants:**

- Exactly **one `IAgStkObjectRoot`** is constructed at app startup; every tree-node, property panel, and toolbar action flows through it.
- Both ActiveX controls (`AxAgUiAxVOCntrl` 3D, `AxAgUiAx2DCntrl` 2D) are `SetAGISTKObject(root)`-bound to the same root, so scenarios appear in both views automatically.
- There is **no REST/WebSocket boundary** inside MVP4. Property-panel changes write directly to `IAgStkObject` properties via COM.
- STK 12 Engine runs as an out-of-process COM server (already installed for MVP1/3).
- Scenarios persist as STK's native `.sc` and `.vdf` files — no custom schema.

**STK binding:** MVP4 uses the C# primary interop assemblies for STK 12 (`AGI.STKObjects.dll`, `AGI.STKUtil.dll`, `AGI.STKVgt.dll`, `AGI.Ui.Controls.dll`) plus the `AxAGI.*` ActiveX wrappers generated by `aximp.exe`. Same COM type library `agi.stk12` wraps in Python — just exposed through C# strong types rather than `win32com` late binding. Keeps MVP4 on the same STK 12 seat MVP1 and MVP3 already require.

## 4. UI layout & authoring flow

```
┌──────────────────────────────────────────────────────────────────────────┐
│  File  Edit  Compute  View  Help                                         │
├──────────────┬─────────────────────────────────────────┬──────────────────┤
│              │ [3D Globe] [2D Map]                     │                  │
│ ▼ Scenario34 │ ┌─────────────────────────────────────┐ │ Aircraft         │
│   ▼ Aircraft1│ │                                     │ │                  │
│     ▶ Sensor1│ │      Embedded STK 3D globe          │ │ Name: Aircraft1  │
│   ▼ Facility1│ │      (AxAgUiAxVOCntrl)              │ │ Color: [cyan ▼]  │
│   ▼ AreaT1   │ │                                     │ │                  │
│   ▼ Coverage1│ │                                     │ │ Route: GreatArc  │
│     ▶ FOM1   │ │                                     │ │ Waypoints:       │
│              │ │                                     │ │  #1 34.2, 74.5   │
│              │ └─────────────────────────────────────┘ │      1500 m      │
│              │ ◄── STK native timeline scrubber ──►    │  #2 36.7, 77.1   │
│              │ ◄── play/pause/step (STK toolbar)  ──►  │      1500 m      │
│              │                                         │  [+ Add Point]   │

├──────────────┴─────────────────────────────────────────┴──────────────────┤
```

### 4.1 Three-pane shell

1. **Object tree (left, ~180 px).** Mirrors STK's scenario hierarchy. Each node wraps an `IAgStkObject`. Right-click menu = `Add Child > Aircraft / Facility / AreaTarget / CoverageDefinition` at the scenario root; `Add Sensor` on Aircraft or Facility; `Add FigureOfMerit` on CoverageDefinition; `Delete` on any node except root.

2. **Center — embedded STK display + native timeline.** Tab-switched `AxAgUiAxVOCntrl` (3D) and `AxAgUiAx2DCntrl` (2D), hosted in WPF via `WindowsFormsHost`. Both tabs bind to the *same* root, so the scenario shows up in both views automatically. Below the globe is STK's native animation toolbar (play / pause / step / reverse) and native time slider — re-exposed in MVP4's chrome; not rebuilt.

3. **Property panel (right, ~260 px).** Content switches on tree selection. Shared shell: header (entity name + type icon), body (type-specific fields), footer (Apply / Reset). Each panel is a WPF `UserControl` bound to a view-model wrapping the underlying COM object. **Edits commit on `Apply` only** — never on keystroke, because many STK setters are expensive and some require a specific assignment order.

### 4.2 Authoring flow (end-to-end)

1. `File > New` → dialog (name, start / stop time) → empty scenario root appears.
2. Right-click root → `Add Aircraft` → name dialog → aircraft at default position. Select in tree → property panel shows aircraft form. Enter waypoints in a grid (lat/lon/alt/speed). `Apply` → view-model writes `IAgAircraft.Route` waypoints → STK redraws track live.
3. Right-click Aircraft → `Add Sensor` → conic half-angle + pointing mode (Fixed az/el or Targeted at …). Cone appears live.
4. `Add Facility` → lat/lon/alt → icon appears.
5. `Add AreaTarget` → polygon vertex grid (lat/lon rows). STK draws the polygon.
6. `Add CoverageDefinition` → grid spec (bounds or attached AreaTarget, resolution), asset list, constraints (altitude min, elevation-angle min). `Add FigureOfMerit` under it → FOM type + statistic.
7. Toolbar `Compute` → runs `IAgCoverageDefinition.ComputeAccesses()` on every CoverageDefinition in the scenario. STK colours the FOM grid natively.
8. Operator scrubs STK's native time slider → aircraft animates, sensors stay attached, access intervals highlight.
9. `File > Save` / `Save As` → writes `.sc` or `.vdf` via `IAgStkObjectRoot.SaveAs(path)` / VDF Connect Command.

### 4.3 Why we don't rebuild STK's timeline

The ActiveX control ships with animation playback bound to STK's own `AnimationTime`. Rebuilding a WPF timeline would duplicate the time controller and risk drift from what STK actually renders. MVP4 exposes the existing STK buttons via its chrome for visual consistency but the events flow through STK.

## 5. Entity catalog

Each tree entity maps 1-to-1 to an STK COM class and one WPF property panel. Fields marked * are required. Defaults match the reference CZML.

| Tree node | STK COM class | Adds under | Property-panel fields | STK renders on add |
|---|---|---|---|---|
| Aircraft | `IAgAircraft` | Scenario root | Name*, Color, Route-type (=Great Arc), **Waypoints grid** (lat\*, lon\*, alt\*, speed), Path visibility, Label | 3D icon + track |
| Facility | `IAgFacility` | Scenario root | Name*, Color, Lat*, Lon*, Alt*, Label | Ground icon |
| AreaTarget | `IAgAreaTarget` | Scenario root | Name*, Outline color, Fill (on/off), Fill color, **Vertices grid** (lat\*, lon\*) | Polygon outline |
| Sensor | `IAgSensor` | Aircraft, Facility | Name*, Pattern (=Simple Conic), Outer half-angle*, Inner half-angle, Pointing (Fixed az/el *or* Targeted at …), Color, Show intersection | Cone attached to parent |
| CoverageDefinition | `IAgCoverageDefinition` | Scenario root | Name*, Grid (lat/lon bounds with steps *or* attached AreaTarget), Assets (multi-select existing objects), Constraints (min altitude, min elevation angle), Access type (=Simple) | Bare grid points (no color until Compute) |
| FigureOfMerit | `IAgFigureOfMerit` | CoverageDefinition | Name*, Type (Access Duration / Revisit Time / Coverage / Sample Count), Statistic (Max / Min / Avg / Total) | Visible after Compute — colored grid |

### 5.1 Common authoring behaviour

- Tree `Add Child` → lightweight name-only dialog → STK object created with sane defaults → tree node added → property panel opens, populated from the new STK object.
- Edits commit on `Apply` only. Tab-away does not commit. `Reset` re-reads from COM.
- Delete → confirmation → `IAgStkObjectRoot.CurrentScenario.Children.Unload(name)`.

### 5.2 Compute

One toolbar action. Iterates over every `CoverageDefinition` under the scenario root and calls `ComputeAccesses()` sequentially. Disabled if no CoverageDefinition exists. FOM values update after compute; STK re-colours the grid natively.

### 5.3 Deliberate exclusions from the entity model

- No satellite / ground-vehicle / missile / ship / comm-object types.
- No sensor pattern types other than Simple Conic — rectangular / custom-pattern not exposed.
- No sensor pointing modes beyond Fixed (az/el) and Targeted.
- No propagator choice — Aircraft routes are Great Arc only.
- No analytic constraint set beyond altitude-min + elevation-angle-min on CoverageDefinition.

## 6. Persistence & lifecycle

### 6.1 Scenario persistence — STK native files

MVP4 does not invent a scenario format. `IAgStkObjectRoot.SaveAs(path)` writes a `.sc` file plus a sibling folder containing `.ac` (aircraft), `.fa` (facility), `.at` (area target), `.sn` (sensor), `.cv` (coverage), `.fm` (fom). These are STK's own formats; STK Insight / STK Desktop can open them as-is. Round-trip compat with STK Desktop is a deliberate MVP4 property — lets the user sanity-check MVP4's output by opening it in Insight.

- **Default location:** `C:\EWTSS\mvp4\scenarios\<name>\<name>.sc` — one folder per scenario.
- **File > Open:** file-type filter has three entries:
  - `STK Scenario (*.sc)` → `root.LoadScenario(path)`
  - `STK Packaged Scenario (*.vdf)` → `root.LoadVdf(path, password)`
  - `All STK Files (*.sc;*.vdf)` (default)
- **VDF password handling:** first attempt uses empty password. If STK throws a password-required error, modal dialog prompts the user; 3 incorrect attempts cancels the open.
- **File > Save As:** two format filters:
  - `STK Scenario (*.sc)` → `root.SaveAs(path)` (folder-based format).
  - `STK Packaged Scenario (*.vdf)` → Connect Command `VDF * Scenario/{name} Save "path" Password "pwd"`. Password optional; empty = unprotected.
- **VDF round-trip:** loading a `.vdf` unpacks into a temp working directory owned by STK. `Save` writes back to the temp copy; only `Save As > VDF` re-packages to a portable `.vdf`. Status bar shows the source path and a `(packaged)` tag.
- **Dirty flag:** single `bool _isDirty` flipped on any `Apply` or any tree Add/Delete. Close/New/Open prompts "unsaved changes" if dirty.

### 6.2 App startup sequence

1. App starts → splash.
2. Construct `AgStkObjectRootClass` (throws if STK 12 Engine isn't installed or COM-registered — caught, error dialog, app exits).
3. Call `root.NewScenario("Untitled")` so the globe has something to bind to.
4. Instantiate both ActiveX controls in the main window; each `.SetAGISTKObject(root)`.
5. Wire `root.OnScenarioNew` / `OnScenarioLoad` / `OnScenarioClose` events to "rebuild tree" handlers.
6. Splash closes; main window visible; operator sees an empty scenario on the globe.

### 6.3 App shutdown sequence

1. `MainWindow.Closing` → if dirty, prompt save.
2. Disconnect ActiveX controls (`SetAGISTKObject(null)`) — prevents controls from holding COM refs across disposal.
3. `root.CloseScenario()` → `Marshal.ReleaseComObject(root)` → `GC.Collect()` + `GC.WaitForPendingFinalizers()`.
4. `Environment.Exit(0)` as a guardrail — STK COM teardown sometimes hangs. 5-second watchdog thread calls `Environment.Exit` if graceful shutdown hasn't completed (same pattern MVP3 used for uvicorn shutdown).

### 6.4 Error-handling matrix

| Class of failure | Surfaces at | Handling |
|---|---|---|
| STK 12 not installed / COM reg missing | `AgStkObjectRootClass` ctor | Error dialog → exit. Installer pre-check. |
| STK license denied at runtime | Ctor or first COM call | Error dialog with license-server hint → exit. |
| `LoadScenario` / `LoadVdf` on corrupt file | File > Open | Dialog with STK's error message; tree stays on previous scenario. |
| `.vdf` password required / incorrect | File > Open | Password prompt dialog; 3 tries then cancel. |
| `ComputeAccesses` throws | Toolbar Compute | Status-bar red text + toast with STK error; tree and globe unchanged. |
| Property `Apply` throws (bad lat/lon, etc.) | Property panel footer | Inline validation message above Apply; COM state untouched. |
| ActiveX control creation fails | App startup | Error dialog → exit. Usually means `AGI.Ui.Controls.dll` not COM-registered. |

### 6.5 What MVP4 does NOT do

No crash recovery, no auto-save, no scenario-file version migration, no concurrent-edit detection (single user, single process), no multi-scenario VDF libraries (STK supports bundles; MVP4 loads the first scenario only), no VDF file-format upgrade.

## 7. Host stack, testing, explicit exclusions

### 7.1 Host stack

| Layer | Choice | Why |
|---|---|---|
| Language | C# | Required; STK's COM API was designed for .NET |
| Runtime | .NET 8 (Windows desktop) | Current LTS; WPF + WinForms in one project; ActiveX hosts cleanly via `WindowsFormsHost`. Fallback to .NET Framework 4.8 only if `aximp`-generated wrappers don't bind in .NET 8. |
| UI | WPF + XAML | Data-binding for property panels; hosts STK's WinForms ActiveX controls via `WindowsFormsHost` |
| MVVM | `CommunityToolkit.Mvvm` | `[ObservableProperty]` / `[RelayCommand]` source generators — minimal panel boilerplate |
| STK interop | `AGI.STKObjects.dll`, `AGI.STKUtil.dll`, `AGI.Ui.Controls.dll` + `AxAGI.*` wrappers (STK 12's `.NET/` folder) | Already present if STK 12 is installed |
| Logging | Serilog (rolling file + debug sink) | Post-mortem trail |
| DI | `Microsoft.Extensions.DependencyInjection` | Only for long-lived services (`StkRootService`, `PersistenceService`) |
| IDE / build | Visual Studio 2022 | Best `aximp.exe` tooling |
| Tests | NUnit + FluentAssertions | Standard .NET |

### 7.3 Testing strategy

| Tier | Scope | STK required? | Runs where |
|---|---|---|---|
| Unit | View-model validation, lat/lon parsing, dirty-flag logic, file-filter behavior, dialog flow, error formatting | No | Dev box + CI |
| Integration | Each entity created via COM → tree; Save/Reload `.sc` + `.vdf`; Compute runs without error; round-trip object-count diff; shutdown leaves no orphan `Agi.Stk12.exe` | **Yes** | Dev box only — CI has no STK |
| Acceptance (manual) | Author a scenario matching `Scenario34.czml`'s topology; save; reopen in STK Desktop / Insight; renders identically | Yes | Dev box only |

**No mock STK.** Unlike MVP1/2/3 (which had `MockStkService`), MVP4's core value *is* the embedded globe. Dev boxes without STK can exercise only Unit-tier code and static UI layouts. Conscious trade-off for a smaller surface.

**CI plan:** GitHub Actions (or equivalent) runs `dotnet build` + Unit tier. Integration tier is a pre-merge gate on a developer's local machine; test output pasted into the PR description.

### 7.4 Acceptance criteria for MVP4

1. Operator authors a scenario from scratch containing one of each of the 6 entity types.
2. Operator saves and reloads as both `.sc` and `.vdf`.
3. Operator opens the attached `Scenario34.czml`-equivalent `.sc` / `.vdf` and sees it render.
4. Operator clicks Compute; STK natively colours the FOM grid on the globe.
5. Any `.sc` / `.vdf` MVP4 writes opens cleanly in STK Desktop / Insight.
6. App shutdown leaves no orphan `Agi.Stk12.exe` after 5 s.

### 7.5 Consolidated explicit exclusions

- No Python, FastAPI, Kafka, SQLite, or any server-side component
- No multi-user / RBAC / login
- No live DRS data overlays, no Kafka subscribers
- No FOM heatmap export (STK renders natively)
- No satellite / ground-vehicle / missile / ship / comm-object types
- No sensor patterns other than Simple Conic
- No propagator types other than Great Arc
- No mock STK — dev needs STK 12
- No cross-platform — Windows desktop only
- No browser delivery, no PDF export, no scenario-catalog UI, no auto-save

## 8. Relationship to MVP1–MVP3 and Option D2

MVP4 stands alongside MVP1/2/3 in the same repo — it does not modify them. Directory is `mvp4/` at the repo root, next to `mvp/`, `mvp2/`, `mvp3/`. The STK 12 seat is shared: MVP1, MVP3, and MVP4 all bind to the same installed engine (not simultaneously — only one backend at a time).

MVP4 is the concrete, buildable form of **Option D2** from [`v2-tech-stack-archive.md`](./v2-tech-stack-archive.md) §14.3. That document left Option D open for team/customer discussion; MVP4 commits to building the smallest useful slice of D2 for evaluation.

If MVP4 succeeds, the natural extensions are:

- **MVP4.1** — scenario-catalog UI (SQLite-backed list of scenarios, thumbnails, tags), PDF export, Windows auth.
- **MVP4.2** — live DRS data overlays via Kafka subscriber + WPF canvas overlay on the STK globe (the risky piece flagged in prior design doc §22.2).
- **MVP4.3** — broader entity set (satellites, comm objects, more sensor patterns) if operator workflows demand it.

None of these extensions are in MVP4's scope. They are listed only so MVP4's boundaries are unambiguous.

## 9. Open questions (deliberately deferred)

These are acknowledged as real questions that MVP4 does **not** answer — they're for MVP4.1+ or a separate design cycle:

1. **Installer & deployment.** MVP4 is built as `dotnet publish` output for dev-box runs only. Production-grade installer (MSI, STK license pre-check, shortcut creation) is out of scope.
2. **Crash recovery / auto-save.** The dirty-flag + save prompt covers planned shutdown only. Process crash mid-edit loses changes.
3. **Scenario sharing / collaboration.** `.vdf` export exists but there is no central scenario library or versioning story.
4. **Performance at scale.** MVP4 is validated against the 6-entity reference scenario. STK handles orders of magnitude more; MVP4's UI (tree-rebuild, property-panel responsiveness) is not stress-tested beyond that.
5. **Accessibility / localisation.** English-only, standard keyboard/mouse only.

These are flagged so the reader knows they were considered, not ignored.
